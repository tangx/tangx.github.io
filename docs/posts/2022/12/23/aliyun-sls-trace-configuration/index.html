Go 使用 OpenTelemetry Collector Contrib 配置并接入 Aliyun SLS Trace 服务

阿里云提供了一个 SLS Trace 服务。 类似于 Jaeger， 可以提供服务的观测性。 入口： 日志服务 SLS -> Trace 服务，

https://sls.console.aliyun.com/lognext/trace

创建新实例之后， 可以看到如下图

其中标记的 1，2，3 之后 授权 和 访问 需要用到的。

除了这几个参数之外， 还需要 logstore 名称， 可以理解为这个就是数据的真实位置。

创建 RAM 账号授权策略

经过测试 SLS Trace 服务接入的时候， 并不支持 STS Token 授权。 必须要单独创建 RAM 账户， 并拿到编程的 Access ID 和 Access Key。

在创建 RAM 账户的时候， 还是要考虑最小化权限的问题。 使用阿里云给的模版

https://help.aliyun.com/document_detail/89676.htm?spm=a2c4g.11186623.0.0.34bd4889YJ1IfX#section-vtv-ccl-z2b

在文档中可以看到， 权限按照功能分成了几个模块。 合并后的最终结果如下。

创建 RAM 账号后， 可以获得编程用的 Access ID 和 Access Key

接入 SLS Trace 服务

SLS Trace 服务的相关文档, 可以参考

https://help.aliyun.com/document_detail/208890.html

在 接入数据概览 中可以看到 ， SLS Trace 服务提供了两种方式接入。

https://help.aliyun.com/document_detail/208894.html

  1. 直接接入: app -> SLS-Trace-Service
  2. 通过 OpenTelemetry Collector 转发接入: app -> OpenTelemetry Collector -> SLS-Trace-Service

直接接入

以 Go 为例， 我们使用 OpenTelmetry 官方的 Fibonacci 案例。

https://opentelemetry.io/docs/instrumentation/go/getting-started/

按照阿里云文档案例， 使用阿里云官方的 sls/provider 库

https://help.aliyun.com/document_detail/208902.htm?spm=a2c4g.11186623.0.0.5bb228f7SnRE3J#task-2058210

1
2
3

import (
    "github.com/aliyun-sls/opentelemetry-go-provider-sls/provider"
)

注意， 当前(2022-12-23) 阿里云提供的 provider 库并不支持最新的 OpenTelemetry v1.11.2 版本。 需要在 go.mod 中将 OpenTelemetry 降级到 v1.10.0。

转发接入

就是使用 OpenTelemetry Collector Contrib 作为中间人转发数据， 好处是 收敛了 RAM 权限账号分发管理。 将 Dev 的配置操作转移到的 Ops ， 减少对业务的侵入。

下载地址: https://github.com/open-telemetry/opentelemetry-collector-releases/releases/tag/v0.68.0

参考阿里云的文档， 按照说明办事， 还是很顺利的。

https://help.aliyun.com/document_detail/208915.htm?spm=a2c4g.11186623.0.0.5bb24889a7qKkl#section-v0u-xbw-har

阿里云这里给出的是一个很基础的配置。

注意， 配置 endpoint 字段的时候， 地址 不需要 配置 project_name 和 端口。 这个配置走的是 http(s) 协议。

receivers:
  otlp:
    protocols:
      grpc:
        endpoint: "0.0.0.0:55680"
      http:
        endpoint: "0.0.0.0:55681"
exporters:
  logging/detail:
    loglevel: debug
  alibabacloud_logservice/traces:
    endpoint: "cn-shanghai.log.aliyuncs.com"
    project: "${project_name}"
    logstore: "${log_store}-traces"
    access_key_id: "${access_key_id}"
    access_key_secret: "${access_key_secret}"
  alibabacloud_logservice/metrics:
    endpoint: "cn-shanghai.log.aliyuncs.com"
    project: "${project_name}"
    logstore: "${log_store}-metrics"
    access_key_id: "${access_key_id}"
    access_key_secret: "${access_key_secret}"
  alibabacloud_logservice/logs:
    endpoint: "cn-shanghai.log.aliyuncs.com"
    project: "${project_name}"
    logstore: "${log_store}-logs"
    access_key_id: "${access_key_id}"
    access_key_secret: "${access_key_secret}"

service:
  pipelines:
    traces:
      receivers: [otlp]           #接收端配置为otlp。
      exporters: [alibabacloud_logservice/traces]   #发送端配置为alibabacloud_logservice/traces。
      # for debug
      #exporters: [logging/detail,alibabacloud_logservice/traces]
    metrics:
      receivers: [otlp]
      exporters: [alibabacloud_logservice/metrics]
    logs:
      receivers: [otlp]
      exporters: [alibabacloud_logservice/logs]

在测试的时候， 这里可以使用 OpenTelemetry Collector 官方的测试用例。

https://github.com/open-telemetry/opentelemetry-go/tree/main/example/otel-collector

注意 阿里云给出的转发配置， 端口给出的是 grpc://0.0.0.0:55680 和 http://0.0.0.0:55681。 而测试用例连接的是 grpc://localhost:30080 ， 需要做相应修改。

其他没什么需要注意的。
